## 一、项目一句话
本项目是一个“未知协议自动逆向原型系统”：输入 pcap/pcapng，自动做预处理、工具选择、字段切分、语义推断、融合，最后输出结构化 JSON 和可读报告。

## 二、背景与要解决的问题
传统协议逆向的痛点是：

1. 高度依赖人工经验，效率低。
2. 不同工具输出格式不统一，难整合。
3. 过程不可复现，难答辩、难比较。
4. 流量复杂时容易“卡在某一步”。

本系统要解决的是：

1. 把逆向流程自动化。
2. 用统一数据模型串联各阶段。
3. 让多工具协同，并带失败降级。
4. 输出“可解释中间结果”，不是黑盒结论。

## 三、系统是怎么运作的（端到端）
从命令开始：python app/main.py --pcap ... --output ...

执行链路是：

1. CLI 入口 app/main.py 把调用转到包内入口。
2. 参数解析和 .env 加载在 src/testcrewai/main.py。
3. 启动 Flow protocol_reverse_flow.py。
4. 依次执行 6 个阶段：
    * preprocess
    * tool_selection
    * segmentation
    * semantic_inference
    * fusion
    * report
5. 每一步写中间 JSON 到输出目录。
6. 任何一步失败都尽量降级，不让全流程崩掉。
7. 最终产出 traffic_profile.json / execution_plan.json / segment_candidates.json / semantic_candidates.json / final_schema.json / report.md / run.log。

## 四、系统里有几个智能体、几个工具
智能体一共 6 个（在 agents.yaml）：

1. preprocess_agent
2. tool_selector_agent
3. segmentation_agent
4. semantic_inference_agent
5. fusion_agent
6. report_agent

任务定义在 tasks.yaml

工具层核心有 5 个：

1. TsharkTool（预处理辅助）
2. NetzobTool（字段切分）
3. NemesysTool（字段切分）
4. NetPlierAdapter（语义推断）
5. BinaryInfernoAdapter（语义推断）

统一子进程执行器：
shell_runner.py

工具总封装：
protocol_tools.py

## 五、项目目前的环境（根据当前工程实际）
当前可确认的运行环境是：

1. 系统 Python：3.8.10
2. 项目虚拟环境 .venv：Python 3.11.15
3. 依赖版本：crewai 1.11.0、scapy 2.7.0
4. 工具独立环境路径（来自 .env）：
    * NETZOB_PYTHON_BIN=/root/venv/bin/python
    * NEMESYS_PYTHON_BIN=/root/venv_nemesys/bin/python
    * NETPLIER_PYTHON_BIN=/root/venv_netplier/bin/python
    * BINARYINFERNO_PYTHON_BIN=/root/venv_binaryinferno/bin/python

也就是说：
项目是“项目主环境 + 每个工具单独 Python 环境”的架构，不是把全部工具塞到一个环境里。

## 六、关键技术点

1. Flow 负责编排，不把逻辑塞进提示词。
2. 多工具统一为 JSON 中间格式，解决格式碎片化问题。
3. 工具调用全走 subprocess，支持 timeout、stderr 捕获。
4. 官方路径优先，失败再降级（保证可运行）。
5. 融合阶段有证据评分和冲突裁决，不是简单拼接。
6. --use-llm 是“可选增强注释”，核心流程主要是规则+工具，不是纯 LLM 黑盒。

src/testcrewai/crew.py：CrewAI @CrewBase，定义 6 个 agent/task。

#### **src/testcrewai/config/**

protocol_reverse_flow.py：主流程编排核心。

## 八、相比传统协议逆向的优点

自动化流程，减少手工重复劳动。
每步有中间 JSON，可追溯、可解释。
工具失败可降级，鲁棒性更高。
多工具协同而不是单工具“赌运气”。
更适合答辩展示“工程化能力”和“系统设计能力”。
