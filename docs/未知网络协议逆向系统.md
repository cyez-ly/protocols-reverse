# 未知网络协议逆向系统

各位老师好，我的本科毕业设计题目是**《==基于多智能体协同的未知网络协议结构逆向推断系统设计与实现==》**。
我的项目使用 **Python + CrewAI** 实现，目标是构建一个**未知协议自动逆向原型系统**：用户输入一份网络流量文件，也就是 **pcap 或 pcapng**，系统自动完成预处理、逆向工具选择、字段切分、语义推断、多结果融合，并最终输出分析报告。

## 一、项目背景与意义

先说一下这个课题为什么值得做。

在网络空间安全领域，随着物联网、工业互联网、专有云服务以及新型恶意软件的发展，现实网络中出现了越来越多**没有公开协议文档**、或者**协议格式不透明**的通信流量，也就是“未知协议”。当前网络环境中约有 **30%** 的流量来源于未知协议，同时超过 60% 的新型恶意软件会利用私有协议的隐蔽性来规避传统检测。这说明，未知协议已经成为网络安全分析中的一个重要难点。

传统做法主要依赖安全分析人员手工看报文、看十六进制数据、反复比对和猜测字段含义。这种方式有几个明显问题：

* 依赖人工经验，效率低；
* 结果不稳定，不同人分析同一个流量，结论可能不一样；
* 难以规模化处理，面对大量复杂流量时很难持续工作；
* 面对二进制协议、变长字段、混淆流量时，经常会“卡住”，很难继续推进。

从研究现状看，现在已经有很多协议逆向方法，例如基于**序列比对**、**统计特征**、**概率推理**，甚至**深度学习**的方法；也出现了一些代表性工具，比如 ==**Netzob**==、==**FieldHunter**==、==**NetPlier**==、==**BinaryInferno**== 、==**BPREN**== （武大网安2025）等。

但这些方法和工具也存在不足：有的擅长字段边界识别，但不擅长语义推断；有的语义能力较强，但冲突处理比较固定；还有一些方法可解释性较弱，或者对环境和输入条件要求较高。

<u>现有单体工具之间存在“信息孤岛”问题，不同算法之间难以共享知识、难以形成反馈，更缺乏有效的冲突协调机制。</u>

⭐因此，我这个课题的核心思路就是：**<u>不再依赖单一工具，而是把多个工具组织起来，让多个智能体模拟“专家协作”的方式共同完成未知协议逆向。</u>**:star:

这也是本项目的意义所在。



#### 1. 理论意义

从理论上讲，本项目尝试把**多智能体系统**引入协议逆向工程，不再把协议逆向看成一个单算法问题，而是把它重构为一个**多角色协同决策问题**。
也就是说，我希望推动协议结构推断从“单算法优化”走向“多策略协同融合”，让系统在字段切分、语义推断、冲突消解等关键环节具备更强的可扩展性和可解释性。

#### 2. 实践意义

从应用上讲，这样的系统可以为**未知协议识别、恶意流量检测、网络行为分析、工控流量监测**等任务提供更自动化、更稳定的技术支持。
它的价值不只是“能跑出一个结果”，更重要的是它能够输出**中间过程和中间证据**，让分析结果可追溯、可解释，而不是一个纯黑盒判断。



## 二、系统架构与整体流程📍

下面介绍一下我的系统是怎么设计的。

我这个系统的总体定位是一个**端到端的未知协议自动逆向原型系统**。

它的**输入**是：

- 一份或多份网络流量文件，格式为 **pcap 或 pcapng**

它的**输出**包括 6 类结果：

- **traffic_profile**：流量画像
- **execution_plan**：工具选择计划
- **segment_candidates**：字段切分候选结果
- **semantic_candidates**：语义推断候选结果
- **final_schema**：最终协议结构模板
- **report.md**：面向分析者的分析报告



#### 1. 整体架构

整个系统采用的是 **Flow + Agents + Tools** 三层架构。

##### 第一层：Flow 层

==（**preprocess → tool_selection → segmentation → semantic_inference → fusion → report**）==

这一层负责定义整个任务的执行顺序。

我没有把所有逻辑都塞进大模型提示词里，而是用流程把系统分成 6 个阶段：

1. **preprocess**
2. **tool_selection**
3. **segmentation**
4. **semantic_inference**
5. **fusion**
6. **report**

也就是说，系统不是一句话丢给大模型让它自由发挥，而是有明确阶段、有明确输入输出的流程式执行。



##### 第二层：Agents 智能体层

为了模拟“专家分工”，我设计了 6 个智能体：

- **Preprocess Agent**：负责预处理
- **Tool Selector Agent**：负责工具选择
- **Segmentation Agent**：负责字段切分
- **Semantic Inference Agent**：负责语义推断
- **Fusion Agent**：负责结果融合
- **Report Agent**：负责报告生成

每个智能体都有自己独立的职责。



##### 第三层：Tools 工具层

这一层是真正执行具体逆向分析的工具集合。当前整合了 4 个主要逆向工具和 1 个预处理工具：

- **Netzob**：用于字段切分
- **NEMESYS**：用于字段切分
- **NetPlier**：用于语义推断
- **BinaryInferno**：用于语义推断
- **tshark**：用于预处理辅助

这些工具并不是直接硬编码在系统里，而是通过统一适配层封装起来，统一输出为 json。



#### 2. 6个阶段流程详细说明

下面我按照系统真实执行流程，详细说明输入和输出是如何一步步变化的。

##### 阶段一：preprocess 预处理

这一阶段输入的是原始 pcap 或 pcapng 文件。
系统首先读取流量，抽取基础信息，例如：

- 报文总数
- 会话信息
- 负载长度分布
- 是否更像文本协议还是二进制协议
- 是否存在明显的端口、协议类型、消息长度特征
- 采样后的代表性 payload

最终形成第一份中间结果：**traffic_profile**。

它可以理解为对输入流量做的一份“体检报告”。

这份结果告诉后续阶段：这份流量大概是什么风格、复杂程度怎样、适合优先调用哪些工具。



##### 阶段二：tool_selection 工具选择

在有了 traffic_profile 之后，系统不会盲目调用所有工具，而是先制定一个**execution_plan**。

这一步主要解决两个问题：

1. 优先使用哪个工具；
2. 如果主工具失败，备选工具是什么。

例如，如果预处理阶段判断这份流量更像结构规整、适合做序列对齐，那么可能优先选 Netzob；

如果字段边界不稳定或者有复杂二进制特征，则可能把 NEMESYS 作为备份；

在语义层面，则根据字段分布和内容特征决定优先使用 NetPlier 还是 BinaryInferno。

所以 **execution_plan** 的本质，就是一份“工具选择方案”。



##### 阶段三：segmentation 字段切分

这一阶段开始真正做协议逆向中的核心工作之一：**把报文切成字段**。

这里的输入主要是：

- 原始流量
- traffic_profile
- execution_plan

这一阶段调用字段切分工具，例如 Netzob 和 NEMESYS，生成多个候选边界方案。

系统不会直接把某个工具的单次结果当成最终答案，而是会做**质量门控**，例如检查：

- 字段边界是否过于碎片化
- 不同报文之间是否存在合理对齐
- 候选字段是否具有可解释的长度模式
- 边界结果是否足够稳定

如果主工具结果质量不够好，系统会自动触发备份工具。

这一步输出的是 **segment_candidates**。也就是说，这里的输出不是“最终字段划分”，而是“多个可比较的字段切分候选结果”。



##### 阶段四：semantic_inference 语义推断

当字段边界基本确定后，系统再进入下一步：推断这些字段**可能代表什么含义**。

例如某个字段可能是：

- 消息类型 type
- 长度字段 length
- 序列号 id
- 时间戳 time
- IP 地址
- 校验类字段 checksum
- 未知控制字段

这一阶段主要通过 NetPlier 和 BinaryInferno 来完成。

如果某些字段推断不明确，系统会保留 unknown，并且根据策略触发备份推断路径。

同时，还会进行边界对齐，避免语义推断和前面的字段分割产生明显冲突。

这一步的输出是 **semantic_candidates**，也就是多个字段语义候选结果。



##### 阶段五：fusion 结果融合

这一步是我认为本系统最体现“多智能体协同”价值的环节。

因为前面两个阶段得到的是：

- 字段切分候选
- 语义推断候选

这些结果之间可能会有冲突，比如：

- 某个工具认为这里应该切分成 3 个字段，另一个认为应该是 2 个字段；
- 某个工具认为该字段是长度字段，另一个认为更像类型字段；
- 有的结果置信度高，有的结果只是弱猜测。

所以在 fusion 阶段，我不是简单把结果拼接起来，而是基于以下信息做综合判断：

- 证据打分
- 工具可靠度
- 结果一致性
- 冲突裁决规则

最终输出 **final_schema**。

这个 final_schema 可以理解为系统对“未知协议结构”的最终机器可读表示。

它通常会描述：

- 每个字段的大致边界
- 每个字段的可能类型或语义
- 整体结构模板
- 每个判断对应的置信信息

通过协调机制解决不同智能体对同一字段分析结果的矛盾问题，最终输出统一的协议规范。



##### 阶段六：report 报告生成

最后一个阶段是把系统中间结果（比如json之类的）整理成**面向人的分析说明**，输出 **report.md**。

这份报告会把整个分析过程串起来，包括：

- 输入文件的基本信息
- 预处理得到的流量画像
- 采用了哪些工具
- 为什么这么选
- 字段切分结果
- 语义推断结果
- 最终 schema
- 可能存在的不确定项

所以这个系统最后不是只输出一个 JSON，而是同时输出**结构化数据**和**可读报告**，既方便后续程序使用，也方便分析人员理解。

## 四、总结

最后总结一下：

我的项目面向的是**未知网络协议自动逆向**这一具有现实价值但又比较复杂的问题。

我提出并实现了一个基于 **Python + CrewAI** 的多智能体协同原型系统。

这个系统以 **pcap/pcapng** 为输入，以 **traffic_profile、execution_plan、segment_candidates、semantic_candidates、final_schema 和 report.md** 为输出，通过 **Flow + Agents + Tools** 的方式，把预处理、工具选择、字段切分、语义推断、结果融合和报告生成组织成一个完整闭环。

我的中期工作重点已经从“方案设计”推进到“系统原型实现”，后续我会继续完善实验验证、性能评估和论文写作，争取形成一个结构完整、结果清晰、可展示、可复现的毕业设计成果。
